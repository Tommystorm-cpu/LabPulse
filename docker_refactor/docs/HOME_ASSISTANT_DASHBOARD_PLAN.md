# Home Assistant Dashboard And Threshold Plan

This note records the planned Home Assistant setup for the Docker refactor.

The goal is to make LabPulse easy to deploy from the repository while keeping the running system controlled from one obvious user-editable file:

```text
~/labpulse-ha/config.yaml
```

## Intended Split Of Responsibility

The Python services should publish facts:

- raw sensor readings
- service health/status
- last known values
- MQTT discovery data

Home Assistant should decide what those facts mean:

- thresholds
- warning states
- dashboard layout
- automations
- notifications
- stale-reading checks

This keeps the Python side focused on reading hardware and publishing data. Home Assistant then acts as the operator interface.

## Repo-Managed Home Assistant Config

It should be possible to include reusable Home Assistant YAML files in the repository, for example:

```text
docker_refactor/
  homeassistant/
    configuration.yaml
    input_numbers.yaml
    templates.yaml
    automations.yaml
    dashboards/
      labpulse.yaml
```

The setup script could copy these files into the live Home Assistant config folder:

```text
~/labpulse-ha/homeassistant/config/
```

The setup script should be careful not to silently destroy an existing Home Assistant setup. A sensible rule would be:

- if the Home Assistant config folder is empty, install the template files automatically
- if it already contains config, preserve it or create backups before replacing anything
- do not track or overwrite Home Assistant's `.storage` folder unless there is a very specific reason

The `.storage` folder is generated by Home Assistant and contains internal UI state. It is more fragile than normal YAML and is not a good first choice for repo-managed configuration.

## Threshold Logic In Home Assistant

Thresholds should live in Home Assistant, not in the Python sensor services.

For example, an air pressure threshold can be represented by an `input_number` helper:

```yaml
air_pressure_minimum:
  name: Air Pressure Minimum
  min: 0
  max: 10
  step: 0.1
  unit_of_measurement: bar
```

Then a template binary sensor can turn that threshold into a warning state:

```yaml
- binary_sensor:
    - name: "Air Pressure Low"
      unique_id: air_pressure_low
      device_class: problem
      state: >
        {{ states('sensor.pressure_monitor_pressure') | float(0)
           < states('input_number.air_pressure_minimum') | float(0) }}
```

This creates a clean split:

```text
Python publishes:       pressure = 6.2 bar
Home Assistant decides: pressure is OK / pressure is low
```

The same pattern can be used for:

- pressure too low
- pressure too high
- flow too low
- temperature too high
- UPS battery low
- service disconnected
- sensor reading stale

## Dashboard Layout

The main LabPulse dashboard should be an operator-style home screen. It should show most of the important information at once, but remain grouped and readable.

A good first layout is:

```text
System Health
  service status
  MQTT/Home Assistant status
  stale readings
  active warnings

UPS / Pi Power
  battery percentage
  voltage
  current
  charge state

Pump Room
  room temperature
  room humidity
  pump pressures
  pump temperatures
  pump flows

Cryogenics / Turbo Pump
  turbo flows
  turbo temperatures

Compressed Air
  air pressure
  pressure threshold
  pressure warning state

Trends
  small graphs for pressure, flow, temperature, and UPS battery
```

Use compact entity cards for most readings. Use gauges only for headline values such as UPS battery or air pressure.

Conditional warning cards should appear only when something needs attention, for example:

- a service status is not `online`
- air pressure is below threshold
- a reading has gone stale

## Dynamic Dashboard Generation

The dashboard can be generated from the same `config.yaml` that generates Docker Compose.

The intended flow is:

```text
~/labpulse-ha/config.yaml
  -> generate_compose.sh
  -> compose.yaml

~/labpulse-ha/config.yaml
  -> generate_homeassistant_config.sh
  -> homeassistant/config/.storage/lovelace
  -> homeassistant/config/packages/labpulse_thresholds.yaml
  -> homeassistant/config/labpulse_alarm_cards.yaml
```

At first, generation can be dynamic by service.

Example:

```yaml
services:
  pressure_monitor:
    enabled: true

  pump_room:
    enabled: true

  turbo_pump:
    enabled: false
```

The generated dashboard would include the pressure monitor and pump room sections, but omit the turbo pump section.

Later, generation can become dynamic by reading. The config can describe the readings each service emits:

```yaml
services:
  pressure_monitor:
    enabled: true
    readings:
      - name: pressure
        label: Air Pressure
        unit: bar
        threshold:
          min: 5.0
```

That allows the generator to create cards, helpers, template sensors, and warnings automatically for each reading.

## Recommended Development Order

1. Keep the current MQTT discovery system working.
2. Manually build a good dashboard in Home Assistant to learn the desired layout.
3. Recreate that layout as YAML in the repository.
4. Add Home Assistant template files to `docker_refactor/homeassistant/`.
5. Update setup so it can install those files into `~/labpulse-ha/homeassistant/config/`.
6. Add a generator that creates dashboard YAML from enabled services in `~/labpulse-ha/config.yaml`.
7. Extend `config.yaml` so each service describes its readings and default thresholds.

## Key Design Principle

The user should not have to hand-edit Docker Compose or dashboard YAML for normal setup.

The ideal long-term workflow is:

```text
edit ~/labpulse-ha/config.yaml
run one setup/generation command
restart Docker Compose
open Home Assistant
```

Everything else should be generated or installed from the repository.
